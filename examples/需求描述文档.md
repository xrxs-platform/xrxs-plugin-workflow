## 一、核心诉求
- 产品形态：身份证号校验插件（后端 Hook）
- 核心价值：为 HR 管理员在员工信息编辑保存环节提供身份证号校验能力，阻止以 0 开头的非法身份证号入库

## 二、需求背景
- 触发场景：HR 管理员在员工信息编辑页面修改员工信息后点击保存，系统触发身份证号校验
- 现有方案：当前缺少对身份证号首位的格式校验，可能导致以 0 开头的错误身份证号被保存入库
- 需求紧迫性：不实施校验会导致脏数据入库，影响后续员工信息统计和合规

## 三、关键角色
- HR 管理员：负责员工信息的维护与编辑，在保存时触发插件校验，当前痛点为缺少身份证号格式校验能力

## 四、用户故事
- HR 管理员：
  - US01: 希望在员工信息编辑保存时，后端自动校验身份证号是否以 0 开头，如果是以 0 开头则阻止保存并提示「身份证号码不能以 0 开头」，以防止错误数据入库。
  - 验收标准：
    - AC1: 员工信息编辑页面保存时，身份证号以 0 开头时，保存被阻止
    - AC2: 阻止保存时，前端提示文案为「身份证号码不能以 0 开头」
    - AC3: 身份证号不以 0 开头时，保存流程不受影响

## 五、范围与约束
- 范围：员工信息编辑页面、保存流程中的身份证号校验
- 不在范围：员工入职登记页面（本期不覆盖）；其他身份证号格式校验规则（如长度、校验位等）
- 技术约束：插件类型为后端 Hook（Java），织入后端保存校验流程
- 外部依赖：宿主系统员工信息保存流程的织入点；前端提示能力
- 待确认：是否存在需要跳过校验的特殊情况（如特殊员工类型、管理员豁免）

## 六、成功标准
- 业务成功标准：HR 管理员在员工信息编辑页面保存时，以 0 开头的身份证号被成功拦截，非 0 开头的正常保存
- 交付完成标准：插件开发完成并通过静态分析、编译、安全扫描，测试环境人工验证通过
